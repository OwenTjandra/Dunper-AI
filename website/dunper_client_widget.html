The Dunper client widget

This is the chat experience paying clients embed on their own site. Unlike the simple iframe popup, it has its own UI so each tenant can theme it to their brand — and every feature of the Dunper chatbot is reachable from inside the panel. Click the bubble in the corner to try the mock flow.

Embed on a client's site

Three lines, anywhere before </body>. The tenantId is what we give them on signup — everything else has a sensible default.

<!-- Dunper AI chat widget -->
<script>
  window.DUNPER = {
    tenantId:     'biz_abc123',         // REQUIRED
    businessName: 'Acme Dental',
    primaryColor: '#FF6B35',            // bubble + outgoing bubble
    accentColor:  '#1E3A8A',            // links + action chips
    greeting:     'Hi! Need help today?',
    locale:       'auto',               // or 'en' / 'id' / 'es' …
    position:     'right',              // or 'left'
    logoUrl:      'https://acme.com/logo.png'
  };
</script>
<script src="https://widget.dunper.com/v1/dunper.js" defer></script>

How a conversation flows

1
Client site loadsThe script reads window.DUNPER, mounts a single floating bubble + dialog, and hydrates any saved session from localStorage.
2
Visitor clicks the bubblePanel slides up. If we don't have a profile yet we show a quick lead form inline as the first card.
3
Visitor types or taps a quick replyWidget POSTs to /v1/messages with tenantId, profileId, text, attachments, locale. Backend routes to the tenant's configured Dunper agent.
4
Bot reply rendersResponse shape: { text, html?, quickReplies?, action?, slots? }. action: 'booking' renders a slot picker; action: 'handoff' shows the human-takeover banner.
5
Booking, escalation, attachmentsEach follow-up calls its own endpoint (/bookings, /handoff, /attachments), but the visitor stays in the same panel. Confirmation comes back inline.
6
Visitor closes the panelState persists. Next time they hit the bubble — even on another page or session — the conversation resumes where they left off.

Backend surface this assumes

Four endpoints + one auth header. The four real network calls in the widget JS are tagged // BACKEND HOOK so the integration is a five-minute job once these exist.

EndpointWhat it does
POST /v1/messagesSend a visitor message. Body: { sessionId, profileId, text, attachments, locale }. Returns: { text, html?, quickReplies?, action?, slots? }.
GET  /v1/availabilityGet bookable slots. Query: serviceId, from, to. Returns: [{ id, label, iso }].
POST /v1/bookingsCreate a booking. Body: { slotId, profileId }. Returns: { ok, confirmation, slot }.
POST /v1/attachmentsMultipart upload. Returns: { url, name, type } for inline display in the conversation.

All four take X-Tenant: {tenantId} as the auth header. Tenant config (agent personality, languages, branding fallbacks, booking services, escalation policy) lives in the Dunper dashboard and is fetched server-side per request — the client widget never sees it directly.

What's a draft, what's real

Already wired

  • All UI structure (bubble, panel, header, messages, composer, toolbar)
  • Quick replies, attachment upload, booking slot picker, lead-form card, handoff banner, language switcher, unread badge
  • localStorage persistence keyed by tenantId
  • Branded theme via window.DUNPER.{primaryColor, accentColor, logoUrl, businessName, greeting, position}

Stubbed (mock data — replace at the // BACKEND HOOK markers)

  • apiSend() returns canned replies based on keywords in the input
  • apiAvailability() returns six made-up time slots
  • apiBook() always succeeds with a random confirmation code
  • apiUpload() uses URL.createObjectURL, doesn't actually upload

Will need real-design pass

  • Header / colour system (currently uses raw primaryColor as a flood)
  • Slot picker layout (currently a 3-column grid — likely wants a date stepper + time list)
  • Lead form (currently three plain inputs — could become a stepper)
  • Empty / error / offline states
  • Sound / desktop notification on incoming bot reply
